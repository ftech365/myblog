## はじめに

XMLやHTMLのファイルを開いたとき、`<タグ>` が入れ子になっていて「何がどこにあるのかわからない」と感じたことはないだろうか。

これは「データ構造」という仕組みを知らないまま見ているからだ。一度この仕組みを理解してしまえば、XML・HTML・JSONのどれを見ても「ああ、あの構造ね」とすんなり読めるようになる。

この記事では、誰もが日常的に使っているパソコンのフォルダを出発点に、データ構造の基本から「型（スキーマ）」「別ファイルの参照」まで順番に解説する。

## そもそも「データ」とは何か

コンピューターが扱う情報のすべてが「データ」だ。

たとえば以下のようなものが全部データにあたる。

- 名前・住所・電話番号
- 商品の値段・在庫数
- アプリの設定値
- Webページの文章や画像

紙のメモに書いた情報も、デジタルに変換すればデータだ。コンピューターはこのデータを読み書きすることで動いている。

---

## データをバラバラに置くと困る

データが少ないうちは問題ないが、量が増えてくると「どこに何があるかわからない」状態になりやすい。

たとえば、デスクトップにファイルを全部置いてしまうと、目的のファイルを探すのに時間がかかる。データも同じで、バラバラに置いたままでは探したり使ったりするのが難しくなる。

そこで「どう整理して入れるか」の仕組みが必要になる。その仕組みのことを**データ構造**と呼ぶ。

難しい話ではない。身近なところに、まったく同じ概念がある。パソコンの「フォルダ」だ。

---

## フォルダで「入れ子」を理解する

フォルダの中にフォルダ、その中にさらにフォルダ……という構造は、誰もが一度は見たことがあるはずだ。

```
📁 仕事                       ← 一番外側（ルート）
├── 📁 2024年                 ← 子フォルダ
│   ├── 📁 上半期             ← 孫フォルダ
│   │   └── 📄 4月報告.docx
│   └── 📁 下半期             ← 孫フォルダ
│       └── 📄 10月報告.docx
└── 📁 2025年                 ← 子フォルダ（2024年と並列）
    ├── 📁 上半期             ← 孫フォルダ
    │   └── 📄 1月報告.docx
    └── 📁 下半期             ← 孫フォルダ
        └── 📄 7月報告.docx
```

この構造にはいくつかのポイントがある。

- 一番外側の「仕事」フォルダが**ルート（根っこ）**で、ここがすべての起点になる
- 「2024年」「2025年」は同じ階層に**並んで存在**できる
- 「上半期」「下半期」は親フォルダの**中にだけ**存在する
- ファイル（`.docx`）は中に何も入れられない**末端**だ

この「入れ子になった階層構造」のことを、コンピューターの世界では**ツリー（木）構造**と呼ぶ。

---

## ツリー図で整理する

先ほどのフォルダ構造をツリー図で表すと、以下のようになる。

```
               📁 仕事（ルート）
              /                \
       📁 2024年            📁 2025年
       /       \             /       \
  📁 上半期  📁 下半期  📁 上半期  📁 下半期
```

木の根っこから枝が広がっていくイメージだ。用語の対応関係は次のとおりだ。

| ツリーの用語 | フォルダで言うと |
|---|---|
| ルート | 一番外側の「仕事」フォルダ |
| 親ノード | 「2024年」「2025年」フォルダ |
| 子ノード | 「上半期」「下半期」フォルダ |
| 葉ノード（末端） | 実際のファイル（.docxなど） |

## フォルダ構造をXMLで書くと

XMLはこのフォルダ構造をそのままテキストで表現したものだ。「フォルダ＝タグ」「ファイル＝タグの中のテキスト」と読み替えると、自然に理解できる。

```xml
<仕事>                              <!-- ルート -->

  <year2024>                        <!-- 子 -->
    <上半期>                        <!-- 孫 -->
      <file>4月報告.docx</file>
    </上半期>
    <下半期>                        <!-- 孫 -->
      <file>10月報告.docx</file>
    </下半期>
  </year2024>

  <year2025>                        <!-- 子（2024年と並列） -->
    <上半期>                        <!-- 孫 -->
      <file>1月報告.docx</file>
    </上半期>
    <下半期>                        <!-- 孫 -->
      <file>7月報告.docx</file>
    </下半期>
  </year2025>

</仕事>
```

> **ポイント**: タグは必ず `<開く>` と `</閉じる>` がセットになる。フォルダが入れ子になるように、タグも入れ子になる。

## HTML・XML・JSONの違い

「入れ子（親・子・孫）」の考え方はHTML・XML・JSONのすべてに共通している。ただし、書き方のルールと用途がそれぞれ異なる。

| フォーマット | 書き方の特徴 | 主な用途 |
|---|---|---|
| HTML | `<タグ>` で囲む。タグの種類が決まっている | Webページの構造を表す |
| XML | `<タグ>` で囲む。タグの名前は自由に決められる | 設定ファイル・データ交換 |
| JSON | `{ }` と `" "` で表現する | APIやアプリ間のデータ転送 |

HTMLの例は次のとおりだ。

```html
<html>
  <body>
    <h1>タイトル</h1>
    <p>本文</p>
  </body>
</html>
```

`<html>` がルート、`<body>` が子、`<h1>` や `<p>` が孫だ。フォルダの構造とまったく同じ考え方になっている。

JSONで同じ構造を表すとこうなる。

```json
{
  "user": {
    "name": "田中",
    "age": 30
  }
}
```

`{ }` の入れ子がフォルダの入れ子に相当する。見た目は違うが、「外側が親、内側が子」という発想は共通だ。

## 「型」という概念：中身のルールを決める

ここまでで「入れ子構造」は理解できた。次に重要なのが**「型」**という概念だ。

### 型とは「中に何が入るかのルール」

先ほどのフォルダ例をよく見ると、`year2024` にも `year2025` にも、必ず `上半期` と `下半期` の2つが入っている。これは「年フォルダには上半期と下半期を必ず入れる」というルールを先に決め、そのルールに従って作られたフォルダ群だからだ。

このような「中身の構造のルール」のことを**型（スキーマ）**と呼ぶ。

### 身近な例で考える

型の概念は日常のいたるところにある。以下にいくつか例を挙げる。

**車の型：**

```
🚗 車
├── フレーム（車体の骨格）
├── エンジン
│   ├── メーカー: Y社
│   ├── 馬力: 150ps
│   └── 排気量: 1500cc
├── ガソリンタンク
│   └── 容量: 50L
├── タイヤ                    ← タイヤ自体にも型がある
│   ├── 前輪（左）            ← さらに子要素を持つ
│   │   ├── メーカー: A社
│   │   ├── サイズ: 195/65R15
│   │   └── 残り溝: 7mm       ← ここが「値」（末端）
│   ├── 前輪（右）
│   │   ├── メーカー: A社
│   │   ├── サイズ: 195/65R15
│   │   └── 残り溝: 6mm
│   ├── 後輪（左）
│   └── 後輪（右）
├── 窓                        ← 窓にも型がある
│   ├── フロントガラス
│   ├── 前列サイド（左）
│   ├── 前列サイド（右）
│   ├── 後列サイド（左）
│   ├── 後列サイド（右）
│   └── リアガラス
└── 内装
```

どんな車でも必ずこれらの要素を持つ。「エンジンのない車」「タイヤのない車」は車として成立しない。中身の構成が型として決まっているのだ。

注目してほしいのは3点だ。

1つ目は、「タイヤ」や「窓」がさらに子要素を持っている点。タイヤという部品にも「前輪・後輪それぞれ左右で4つ」という独自の型がある。**型の中にさらに型が入れ子になっている**構造だ。

2つ目は、「前輪（左）」の中にメーカー・サイズ・残り溝といった項目がある点。これが**ひ孫にあたる要素**だ。

3つ目は、`A社` や `195/65R15` や `7mm` といった具体的な数値や文字列が末端に並んでいる点。これがツリーにおける**「値」**で、フォルダで言えば実際のファイルの中身に相当する。型はあくまで「入れ物の形」を決めるルールであり、値はその入れ物に実際に入るデータだ。

**PCの型：**

```
💻 PC
├── CPU（演算処理）
│   ├── メーカー: X社
│   ├── モデル: Core i7-1360P
│   └── コア数: 12
├── メモリ（一時記憶）
│   ├── メーカー: Y社
│   ├── 容量: 16GB
│   └── 規格: DDR5
├── マザーボード（各部品の土台）
│   ├── メーカー: Z社
│   └── チップセット: H770
├── ストレージ（永続記憶）
│   ├── 種別: SSD
│   ├── 容量: 512GB
│   └── インターフェース: NVMe
└── OS（基本ソフト）
    ├── 種別: Windows
    └── バージョン: 11
```

これらが揃って初めて「PCとして動く」。型が守られていることで、誰が組み立てても同じように機能する。

### XMLでの型の表現

先ほどのXMLに型の考え方を当てはめると、こうなる。

```xml
<!-- 「年フォルダ」の型：上半期と下半期を必ず持つ -->
<year2024>
  <上半期>
    <file>4月報告.docx</file>
  </上半期>
  <下半期>
    <file>10月報告.docx</file>
  </下半期>
</year2024>

<!-- year2025も同じ型に従う -->
<year2025>
  <上半期>
    <file>1月報告.docx</file>
  </上半期>
  <下半期>
    <file>7月報告.docx</file>
  </下半期>
</year2025>
```

どの年のデータを見ても「上半期と下半期がある」と事前にわかるため、データを処理するプログラムが書きやすくなる。

> **ポイント**: XMLでは型を `.xsd` ファイル、JSONでは `schema.json` として別ファイルで定義・管理することが多い。型を外部ファイルで持つことで、複数のデータファイルが同じルールを共有できる。

### 型はアプリ設計にも現れる

プログラムの設計でも同じ発想が使われている。たとえば**MVCモデル**は「WebアプリはModel・View・Controllerの3層で構成する」という型（設計パターン）の一種だ。「このフォルダには何が入るか」を決めることと、本質的に同じ考え方がアプリ設計にも根付いている。

## 応用：複数ファイルに分割して参照する

実際のシステムでは、XMLの設定が1ファイルに収まることはほとんどない。役割ごとに複数のファイルへ分割し、互いに参照し合う構成が一般的だ。

### なぜファイルを分割するのか

設定を複数ファイルに分割する理由は主に以下の3点だ。

- **修正が1か所で済む**：共通の設定を変えたいとき、1ファイルだけ直せばよい
- **役割が明確になる**：「このファイルはDB接続」「このファイルはログ設定」と管理しやすくなる
- **再利用できる**：同じ設定を複数の場所から参照できる

### ファイル間の参照関係

複数ファイルに分割したとき、ファイル同士がどのように参照し合うかを図に示す。

`main-config.xml` が各設定ファイルを `import` で束ねる入口となる。また `database.xml` と `server.xml` は共通の型定義ファイル（`shared-schema.xsd`）を参照しており、同じ型のルールを共有している。

### importで別ファイルを読み込む

XMLでは `import` や `include` を使って「別ファイルの内容をここに読み込む」という指示ができる。

```xml
<!-- main-config.xml：メインの設定ファイル -->
<config>
  <import src="database.xml"/>   <!-- DB接続設定を読み込む -->
  <import src="server.xml"/>     <!-- サーバー設定を読み込む -->
  <import src="logging.xml"/>    <!-- ログ設定を読み込む -->
</config>
```

それぞれの設定ファイルは独立したXMLとして管理されている。

```xml
<!-- database.xml -->
<database>
  <host>localhost</host>
  <port>5432</port>
</database>
```

```xml
<!-- logging.xml -->
<logging>
  <level>INFO</level>
  <file>app.log</file>
</logging>
```

`main-config.xml` を読み込むと、importで指定した各ファイルの内容が自動的に取り込まれ、1つの設定として扱われる。ファイルは分かれていても、論理的には1つのツリー構造として機能する点がポイントだ。

> **注意**: 参照先のファイルが存在しない・パスが間違っているとエラーになる。import構文を使うときはファイルの場所に注意すること。

## 用語まとめ

この記事で登場した用語を整理しておく。

| 用語 | 意味 | フォルダで例えると |
|---|---|---|
| ルート | ツリーの一番上。親がいない唯一の大元 | 一番外側の「仕事」フォルダ |
| 親ノード | 自分の中に他のノードを持つ | 「2024年」フォルダ |
| 子ノード | 別のノードの中に含まれる | 「上半期」「下半期」フォルダ |
| 孫ノード | 子の中にある子 | さらに下の階層のフォルダ |
| 葉ノード | 子を持たない末端 | 実際のファイル（.docxなど） |
| タグ | XML・HTMLでデータを囲む記号 | `<year2024>…</year2024>` |
| ツリー構造 | 入れ子の階層構造全体のこと | フォルダ全体の構成 |
| 型（スキーマ） | 「中に何が入るか」のルール定義 | 年フォルダには上半期・下半期が入る、というルール |
| import / include | 別ファイルの内容をここに読み込む指示 | ― |
| 参照 | 実体が別ファイルにあり、そこを指し示すこと | ― |

## まとめ

XML・HTML・JSONに共通する「ツリー構造」は、パソコンのフォルダと同じ入れ子の仕組みだ。

そこに「型（スキーマ）」の概念が加わることで、「このデータ構造には何が入るべきか」というルールが生まれる。車にはエンジンとタイヤ、PCにはCPUとOS、WebアプリにはMVC——いずれも型の考え方が根底にある。

さらに実際のシステムでは、設定を複数ファイルに分割して `import` で参照し合う構成が多い。ファイルは分かれていても論理的には1つのツリーとして機能する、という点を押さえておくとよい。

まずは「フォルダの入れ子」のイメージを土台にして、この記事で紹介した概念を少しずつ積み上げていってほしい。
