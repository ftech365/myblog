## 全体構成

当ブログの技術スタックは以下の4つのモジュールで構成されている。

| モジュール | 役割 |
|---|---|
| Astro v5 | フレームワーク・全体の司令塔 |
| Tailwind CSS v4 | CSSフレームワーク・デザイン担当 |
| Expressive Code | コードブロックの表示強化 |
| Pagefind | サーバーなしの全文検索 |

ホスティングは **GitHub Pages**、デプロイは **GitHub Actions** で自動化している。

---

## 記事が公開されるまでの流れ

記事を書いてから公開されるまでの流れは以下のとおりだ。

```
STEP 01: Markdown
  記事を .md ファイルで書く
      ↓
STEP 02: Astro Build
  静的HTMLに変換・出力
      ↓
STEP 03: Tailwind CSS
  スタイルをCSSに変換
      ↓
STEP 04: Expressive Code
  コードブロックを美しく変換
      ↓
STEP 05: Pagefind
  検索インデックスを自動生成
      ↓
STEP 06: GitHub Pages
  全ファイルを公開・配信
```

実際の作業はMarkdownで記事を書いて `git push` するだけだ。STEP 02〜06はGitHub Actionsが自動で実行する。

## 各モジュールの詳細

各モジュールがどのような役割を担っているかを整理する。

---

### Astro（フレームワーク）

ブログ全体の骨格・エンジンとして機能するフレームワークだ。

主な特徴は以下のとおりだ。

- **ページルーティング** — `src/pages/` のファイル構造がそのままURLになる
- **Markdownの記事管理** — `src/content/posts/` のMDファイルを自動で記事として読み込む
- **静的HTML出力** — JavaScriptをほぼ出力しないため表示が超高速
- **型安全なスキーマ** — Zodで記事のフロントマターをバリデート

Astroの最大の特徴は「デフォルトでJavaScriptを出力しない」点だ。表示速度が重要なブログには最適な選択肢といえる。

---

### Tailwind CSS（CSSフレームワーク）

見た目・デザインの実装を担当するCSSフレームワークだ。

主な特徴は以下のとおりだ。

- **ユーティリティクラス** — `class="bg-gray-900 p-4"` のようにHTMLに直接スタイルを書く
- **ダークモード対応** — `dark:` プレフィックスで切り替えを簡単に実装
- **レスポンシブ対応** — `md:` `lg:` などのプレフィックスで画面サイズ別に調整
- **v4の変更点** — 設定ファイル不要、CSSファイル内に直接設定を記述する方式に変更

v4からは `tailwind.config.js` が不要になり、よりシンプルな構成になった。

---

### Expressive Code（コード表示ライブラリ）

コードブロックの表示を強化するライブラリだ。技術ブログにとって読みやすいコード表示は重要なため採用した。

主な特徴は以下のとおりだ。

- **シンタックスハイライト** — 言語ごとに色付けして読みやすく表示
- **コピーボタン自動追加** — すべてのコードブロックにワンクリックコピーが付く
- **ファイル名・行番号表示** — `title="config.ts"` と書くだけでタブが表示される
- **行ハイライト** — `{3-5}` のように書くと特定行を強調表示できる

Markdownで通常どおりコードブロックを書くだけで自動的に適用される点が便利だ。

---

### Pagefind（検索エンジン）

サーバーなしで全文検索を実現するライブラリだ。GitHub Pagesのような静的ホスティングでも検索機能が使える理由がこれだ。

主な特徴は以下のとおりだ。

- **静的インデックス生成** — ビルド後に `dist/` を解析して検索用ファイルを自動生成
- **サーバー不要** — 生成された静的ファイルだけで全文検索が動く
- **日本語対応** — 日本語テキストの分かち書きも自動処理
- **高速検索** — インデックスが軽量なため検索レスポンスが速い

通常、全文検索にはElasticsearchのようなサーバーサイドの仕組みが必要だ。Pagefindはその制約をビルド時のインデックス生成で解決している。

## モジュール間の連携

4つのモジュールがどのように連携しているかを整理する。

**Astro** は全体を統括する司令塔だ。Tailwind・Expressive CodeはビルドときにAstroが呼び出す構造になっており、`astro.config.mjs` で統合設定を管理している。

**Tailwind CSS** はAstroコンポーネント（`.astro`）内のクラスを読み取り、実際に使われているCSSだけを生成・出力する。未使用のCSSは自動削除されるため、最終的なCSSファイルのサイズが最小限に抑えられる。

**Expressive Code** はMarkdownのコードブロックをAstroがHTMLに変換する際に割り込んで、装飾済みのHTMLに置き換える。Markdownを書くだけで自動適用されるため、記事執筆時に意識する必要はない。

**Pagefind** だけはやや特殊で、Astroのビルドが完了した後に独立して `dist/` フォルダを読み込んで検索インデックスを生成する。他の3つとは独立して動く後処理の位置づけだ。

## この構成を選んだ理由

技術スタックを選定する際に重視した基準は3つだ。

**無料で運用できること** — GitHub Pagesは無料、Astroも無料のOSSだ。個人ブログにコストをかけたくない場合の現実的な選択肢になる。

**書くことに集中できること** — Markdownで記事を書いて `git push` するだけで公開される。デプロイ作業が自動化されているため、記事執筆以外の手間が最小限だ。

**表示が速いこと** — Astroの静的HTML出力により、サーバーサイドの処理が不要になる。PageSpeedスコアが高く保てるため、読者体験とSEOの両方に寄与する。

## 主要な設定ファイル一覧

このブログの設定を確認・変更する際に参照するファイルを整理する。

---

### 重要度：高

設定変更が必要になったときに最初に確認するファイル群だ。

| ファイル | 内容 |
|---|---|
| `astro.config.mjs` | サイト全体の設定。URLベース・プラグイン・ビルド設定など |
| `src/content/config.ts` | 記事スキーマ定義。frontmatterの項目・型・バリデーション |
| `.github/workflows/deploy.yml` | GitHub Actionsの自動デプロイ設定 |

特に `src/content/config.ts` は新しいタグやカテゴリを追加する際に変更が必要になるため、覚えておくと役立つ。

---

### 重要度：中

直接触る機会は少ないが、把握しておくと理解が深まるファイル群だ。

| ファイル | 内容 |
|---|---|
| `package.json` | 使用パッケージとバージョン一覧 |
| `tsconfig.json` | TypeScriptのコンパイル設定 |
| `src/styles/global.css` | サイト全体のCSS変数・デザイントークン定義 |

## まとめ

構成・フロー・各モジュールの役割を図解にまとめた。静的サイトジェネレーターを使ったブログ構築は、サーバー管理が不要で学習コストも低い。「まず動くものを作る」という観点では、この構成は現時点で最もバランスの良い選択肢の一つだと思っている。
